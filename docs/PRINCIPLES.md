# Principles

These principles are the timeless design contract for PhantomForce. They should guide every feature, refactor, automation, UI surface, and worker integration.

## Chat Is Not The Product

Chat is an input surface. PhantomForce is the operating intelligence around the business. A great chat box is useful only if it routes to outcomes, memory, approvals, tools, workers, evidence, and execution.

## Outcomes Over Conversations

The user should not need to babysit a conversation to get business value. A request should become an outcome, draft, decision, asset, automation, approval, or business record when appropriate.

## Signals Create Decisions

Raw data is not enough. PhantomForce must turn signals into clear decisions. If a signal does not affect a decision, priority, automation, memory, or outcome, it should not dominate the UI.

## Evidence Before Opinion

Every intelligence feature should show source, date, confidence, and reasoning boundary. PhantomForce should not pretend guesses are facts.

## Execution Before Generation

Generating text or media is only the beginning. The product value is in preparing work, routing it, storing it, reviewing it, approving it, publishing it when allowed, and proving what happened.

## Departments Over Features

Users think in business departments and responsibilities, not tool names. Leads, media, websites, money, memory, security, analytics, automations, and workforce should feel like operating departments.

## Institutional Memory Over Chat History

Memory should be durable business knowledge, not a pile of transcripts. Temporary history can expire. Important facts, preferences, corrections, and rules should be promoted, editable, and forgettable.

## Automation Should Reduce Friction

Automation should make the next right action easier. It should not create noise, fake work, risky sends, or confusing setup. Automations should have clear parameters, status, history, and owner control.

## Approval Preserves Control

External actions need gates. Sends, posts, uploads, deployments, payments, invoices, destructive edits, credential changes, and client-facing actions require approval unless a future explicit permission model safely allows otherwise.

## The User Should Never Need To Search For Work

Work should arrive. PhantomForce should surface what matters: approvals, follow-ups, blocked items, next best actions, overdue work, system issues, and opportunities.

## Hide The Machinery

Backend tools create the moat, but normal users should not have to think about them. Apify, n8n, provider names, scaffolds, worker internals, and local bridges belong behind the curtain unless the owner/developer surface explicitly needs them.

## Honest Empty States

If no real data exists, say so cleanly. Do not seed fake work, fake clients, fake analytics, fake tasks, or fake worker activity. A brand-new account should feel ready, not fabricated.

## Mobile Is A First-Class Operating Surface

The owner may be on a phone. The app must remain usable when the keyboard opens, when the nav collapses, and when the screen is narrow.

## Alive But Useful

Animation, characters, maps, and graphics should clarify what is happening. They should not distract from work or cover core controls.

## Compound Advantage

Every correction, approval, asset, signal, and outcome should make PhantomForce better next time. The moat is the coordinated system: memory plus tools plus workflow plus evidence plus safe execution.
